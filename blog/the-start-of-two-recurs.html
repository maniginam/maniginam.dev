Why Deliberate Ugliness Matters

The deliberate ugliness part is worth dwelling on. In the TDD world, this is the Green phase -- write the minimum code to make the test pass, no matter how ugly. The instinct to write it "correctly" the first time is strong, but resisting that instinct is the whole point. If you optimize too early, you skip the refactoring step, and the refactoring step is where the real design happens.

Justin and I took this to an extreme for the video, giving each other intentionally terrible solutions -- nested ifs, hardcoded vectors, special-case checks. But there is a genuine lesson buried in the comedy. When you see the ugly version first, the clean version becomes obvious. You cannot refactor toward clean code if you do not have ugly code to refactor from. The ugliness is not a failure; it is the first draft.

Pairing with Justin also highlighted something about different approaches to the same problem. He started with 2, I started with 1. Neither was wrong. But the difference in starting point led to different design decisions at every step. This is exactly why pairing -- and later, code review -- matters so much. You see your own blind spots reflected in someone else’s choices.
